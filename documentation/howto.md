# Рецепты

## Полный прогон с нуля

```bash
./download.py     # сетевая часть: sparse-checkout всех версий в cache/
./analyze.py      # локальная часть: archive + метрики + отчёты + сводка + лендинг
```

## Добавить новую версию YDB

1. Выяснить имя ветки в `ydb-platform/ydb` (например, `stable-26-2`).
2. Добавить запись в `config.json`:
   ```json
   "26.2": {"ref": "stable-26-2", "url_version": "v26.2"}
   ```
3. Запустить:
   ```bash
   ./download.py 26.2
   ./analyze.py
   ```

Никакой код править не нужно. `analyze.py` без аргументов всегда
пересчитывает сводку и лендинг по всем доступным версиям.

## Обновить уже добавленную версию

Ветки `stable-*` иногда получают патчи, а `main` меняется постоянно:

```bash
./download.py --refresh main
./download.py --refresh 26.1
./analyze.py
```

Без `--refresh` `download.py` пропустит уже скачанную папку.

## Запустить только сравнение (без перекачки)

После того как `cache/<v>/docs/` уже на месте:

```bash
./analyze.py 25.3
```

Полезно, если меняешь метрики/веса и хочешь пересчитать без сети.

## Открыть HTML-отчёт в браузере

`analyze.py` всегда пишет рядом и `report.txt`, и `report.html`.

```bash
open docs/index.html             # лендинг с навигацией
open docs/summary.html           # сводка — точка входа по существу
open docs/main/report.html       # детально по версии
xdg-open docs/main/report.html   # Linux
```

В HTML цветной бейдж оценки (от красного `1` до зелёного `10`),
кликабельные RU и EN ссылки, отдельные блоки статистики и
«ожидаемо одноязычных» страниц. В шапке каждого отчёта — строка
«Сгенерировано: YYYY-MM-DD».

## Опубликовать обновлённый отчёт

Все артефакты — в `docs/`, и эта папка коммитится в репозиторий.
GitHub Pages раздаёт её содержимое автоматически.

```bash
./download.py --refresh main
./analyze.py
git add docs
git status                   # глянуть, что изменилось
git commit -m "docs: snapshot $(date +%Y-%m-%d)"
git push
```

Через минуту-две сайт обновится. URL — в Settings → Pages.

Если поменялся только один отчёт и не хочется пересобирать всё:

```bash
./analyze.py 25.3            # пересоберёт 25.3, сводку, лендинг
git add docs/25.3 docs/summary.* docs/index.html docs/meta.json
git commit -m "docs(25.3): refresh $(date +%Y-%m-%d)"
git push
```

## Открыть прошлый снимок

Снимки лежат под `docs/history/<YYYY-MM-DD>/` (см.
[architecture.md → Архив прошлых снимков](architecture.md#архив-прошлых-снимков)).

```bash
open docs/history/index.html                          # список всех снимков
open docs/history/2026-06-01/summary.html             # конкретный снимок
```

Сравнить две сводки можно diff'ом по txt:

```bash
diff docs/history/2026-06-01/summary.txt docs/summary.txt
```

## Пересобрать только сводку

```bash
./scripts/summary.py
```

Полезно, если правил `results.json` руками или поменял палитру.
`analyze.py` зовёт `summary.py` автоматически в конце, так что после
обычного прогона `docs/summary.{txt,html}` уже актуальны.

Что в сводке:

- итоговая таблица (Pages / Avg / Score=1 / Score=10 / One-lang);
- матрица распределения оценок «версия × балл»;
- средняя оценка по разделам «раздел × версия» (видны тренды).

Версии без `results.json` пропускаются с предупреждением — это полезно,
если в конфиге появилась новая версия, но `analyze.py` для неё ещё не
гонялся.

## Пересобрать лендинг

```bash
./scripts/index.py
```

Перечитывает список снимков из `docs/history/` и обновляет
`docs/index.html` + `docs/history/index.html`. Дата прогона — сегодня
(можно передать аргументом, см. `--help`). `analyze.py` зовёт этот шаг
автоматически.

## Принудительно перепаковать архив

Если по какой-то причине нужно вручную переархивировать текущее
состояние:

```bash
echo '{"generated_at": "2026-01-01"}' > docs/meta.json
./analyze.py
```

`archive.py` увидит несовпадение даты и скопирует текущий снимок в
`docs/history/2026-01-01/`. После прогона `docs/meta.json` обновится
сегодняшней датой.

## Поменять цветовую палитру оценок

В `scripts/lib.py` отредактировать словарь `SCORE_COLORS`:
`{балл: (фон_hex, цвет_текста_hex)}`. После правки запустить
`./analyze.py` (или точечно `./scripts/report.py <v>` +
`./scripts/summary.py`) — `download` не нужен. Палитра используется и
в per-version, и в сводном отчёте.

## Посмотреть только проблемные страницы

```bash
grep -E '^\s+([1-7])\s' docs/25.3/report.txt
```

`report.txt` уже отсортирован по возрастанию, поэтому проблемы — сверху.

## Сравнить распределения между версиями

```bash
for v in 25.3 25.4 26.1 main; do
  echo "=== $v ==="
  grep -A20 'Распределение' docs/$v/report.txt
done
```

## Одноязычные страницы

Если есть страницы, которые существуют только на одном языке **по
дизайну** (например, анонсы конкретного выступления на русском или
английском в `public-materials/`), их можно исключить из основного
отчёта: они не должны считаться «отсутствующим переводом».

В `config.json`:

```json
"single_language_patterns": [
  "public-materials/*",
  "blog/podcasts-ru/*"
]
```

Паттерн — это fnmatch-глоб от **относительного пути** страницы
(от `ru/core` или `en/core`). `*` матчит любые символы, включая `/`,
так что `public-materials/*` покрывает любую глубину под этой папкой.

Что произойдёт:

- страница, матчащая паттерн и существующая только на одном языке,
  не попадает в основную таблицу и гистограмму;
- такие страницы выводятся отдельной секцией в конце `report.txt` с
  меткой `[RU-only]` или `[EN-only]`;
- если та же страница вдруг появилась на обоих языках, она
  оценивается обычным образом.

Подробнее об эффекте на оценку — [scoring.md](scoring.md#страницы-вне-оценки).

## Изменить список исключаемых папок

В `config.json`:

```json
"exclude_dirs": ["_includes", "_assets", "_examples"]
```

Это влияет только на список **страниц**, которые считаются единицами
сравнения. Раскрытие includes продолжает работать (см.
[architecture.md](architecture.md)).

## Сменить методологию оценки

Поправить веса или добавить признак — `scripts/lib.py`:
`SCORE_WEIGHTS` и `metrics`. Подробно — [scoring.md](scoring.md).

## Очистить локальный кэш версии

```bash
rm -rf cache/25.3
```

Следующий `./download.py 25.3` скачает заново. `docs/25.3/` при этом
остаётся актуальным — он перезапишется на следующем `./analyze.py 25.3`.

## Закоммитить проект в git с нуля

`.gitignore` уже исключает `cache/`, `__pycache__/`, `.DS_Store`,
`.claude/settings.local.json`. Папка `docs/` коммитится. Безопасно
делать:

```bash
git init
git add -A
git commit -m "Initial commit"
```

Кто клонирует репозиторий — запускает `./download.py && ./analyze.py`
и видит у себя ровно те же отчёты, что лежат в `docs/`.
